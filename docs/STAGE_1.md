# Stage 1 — Core Verification Engine

**Status:** Complete  
**Ship date:** 2026-03-06  
**Stack:** Next.js 14 (App Router) · TypeScript (strict) · Tailwind CSS · SQLite / Drizzle ORM

---

## Overview

Stage 1 establishes the cryptographic foundation and primary ingestion UI for
Blackhorse Sentinel. Every downstream stage (evidence linking, report
generation, enterprise billing, BHL signing) builds on this layer.

The four goals of Stage 1:

1. **Ingest** artifacts from three input types: file upload, URL, raw text
2. **Hash** every artifact with SHA-256 + SHA-3-512 server-side
3. **Timestamp** every artifact with a server-assigned ISO 8601 timestamp
4. **Persist** each artifact record with a unique nonce in SQLite

---

## Cryptographic Design

### Why dual-hash?

| Hash | Length | Purpose |
|------|--------|---------|
| SHA-256 | 256 bits (64 hex chars) | Industry standard; broad tool compatibility |
| SHA-3-512 | 512 bits (128 hex chars) | Quantum-transition hash; feeds BHL encoding at Stage 6 |

**SHA-3-512 is not optional.** At Stage 6, the SHA-3-512 digest is BHL-encoded
and signed with CRYSTALS-Dilithium via the Blackhorse Protocol. Changing or
truncating this digest before Stage 6 integration breaks the signing chain.

For the BHL encoding spec, see `../blackhorse/docs/BHL_SPEC.md`.

### Why server-side only?

Client-computed hashes are untrusted because:
- The browser has no guaranteed access to the original file bytes after transport
- Network interception can substitute content between hash and upload
- Client clocks cannot be trusted for timestamps

All hashing (`lib/crypto/hash.ts`) and timestamping (`lib/crypto/timestamp.ts`)
run server-side in Node.js using the built-in `node:crypto` module.

### Nonce

Each artifact record includes a 16-byte (32 hex char) cryptographically secure
nonce generated by `crypto.randomBytes(16)`. This ensures two identical
artifacts ingested at different times produce distinct records and cannot be
confused in the audit trail.

## API Reference

### `POST /api/artifacts`

Ingest a new artifact.

**File upload** (multipart/form-data):
```
POST /api/artifacts
Content-Type: multipart/form-data

file: <binary>
```

**URL ingestion** (application/json):
```json
{ "type": "url", "url": "https://example.com/doc.pdf" }
```

**Text ingestion** (application/json):
```json
{ "type": "text", "content": "The quick brown fox..." }
```

**Response (201 Created)**:
```json
{
  "success": true,
  "data": {
    "id": "a3f9b2c1-d4e5-6789-abcd-ef0123456789",
    "hashSha256": "a3f9b2c1d4e5f6789abcdef0...",
    "hashSha3_512": "7d8c9e0f1a2b3c4d5e6f789...",
    "timestampIso": "2026-03-06T18:45:32.891Z",
    "sourceType": "text",
    "sizeBytes": 23,
    "mimeType": "text/plain",
    "fileName": null,
    "content": "The quick brown fox...",
    "sourceUrl": null,
    "metadataJson": "{}",
    "nonce": "f0e1d2c3b4a596877869...",
    "status": "unverified",
    "createdAt": "2026-03-06T18:45:32.893Z"
  }
}
```

### `GET /api/artifacts`

List artifacts with pagination.

Query params: `page` (default 1), `pageSize` (default 20, max 100), `status`, `sourceType`

### `GET /api/artifacts/:id`

Retrieve a single artifact by UUID.

### `GET /api/health`

System health check — returns status, artifact count, protocol version.

## Database

SQLite via `better-sqlite3` + Drizzle ORM. Database file: `.data/sentinel.db`  
Schema: `lib/db/schema.ts` (ORM) | `schema.sql` (SQL reference)

**Migration path to PostgreSQL (Stage 4):**
1. Change `drizzle.config.ts` dialect from `sqlite` to `postgresql`
2. Update `lib/db/client.ts` to use `drizzle-orm/node-postgres`
3. Point `DATABASE_URL` at Supabase
4. The repository layer (`lib/db/repositories/`) requires no changes

## What is NOT in Stage 1

Do not build the following until explicitly directed:

- Evidence linking (Stage 2)
- Report generation and PDF export (Stage 3)
- Enterprise orgs, API keys, billing (Stage 4)
- SOC 2, HIPAA, GDPR compliance tooling (Stage 5)
- Blackhorse Protocol BHL encoding + Dilithium signing (Stage 6)
  — requires Protocol Stages 3–6 to be available
- AI agent verification API (Stage 7)
- Public proof explorer (Stage 8)

## Compliance Notes

A Fortune 500 CISO reading this document should know:

1. **No plaintext secrets** are stored. Keys, nonces, and tokens are
   either hashed or stored in environment variables.

2. **Hashing is deterministic and auditable.** The SHA-256 and SHA-3-512
   algorithms used are FIPS-approved. Node.js delegates to OpenSSL 3.x.

3. **Timestamps are server-assigned.** Client clocks are never trusted.
   All timestamps are UTC ISO 8601 with millisecond precision.

4. **The post-quantum transition path is designed in.** SHA-3-512 is
   preserved for BHL → Dilithium integration at Stage 6, in accordance
   with NIST PQC standards and the Blackhorse Protocol specification.

5. **The full schema is available** in `schema.sql` for security review.
